1. 项目定位

SuperModel Router (SMR) 是一个 多 Provider / 多 Key / 智能路由 的 OpenAI-兼容 API 代理. 核心价值: 把"哪个 provider / 哪个 key / 哪个 model" 3 个维度决策完全交给程序, 老大的所有 chat 调用走 http://localhost:6473/v1/chat/completions 一个端点即可, 不感知后端差异.

3 维度路由

Provider (openrouter/freem/newapi/...) → Key (多 key 轮询/401 换) → Model (评分+规则+filter)

OpenAI 兼容

/v1/chat/completions / /v1/embeddings / /v1/models 全部标准, 现有 SDK 0 改动接入

多租户对外 API

v3.7.1+ /v1/public/chat/completions 独立 API key 管理, 不走 admin key 池

上下文感知

v3.4+ 上下文桥接 (切链同步) + v3.8 切链压缩 (按目标 model context_window 自动)

可视化

admin UI i18n (中/英) + 路由表 / 模型列表 / 公开 key / 上下文桥接统计

自升级

v3.1+ /v1/admin/version + GitHub release 轮询 + /v1/admin/upgrade 生成 4 种升级命令

2. 架构

数据流 (典型 chat 场景)

  1. 客户端 POST /v1/chat/completions 带 {model, messages, stream, ...}
  2. openai_routes 接收 → 调 engine.pick_chain 选 candidate 链 (provider+key+model)
  3. context_bridge.compress_for_target 按链中 target context_window 压缩 body
  4. engine.execute_chain 串行试 candidate, 401/5xx 自动切下一个, 流式切时显式 aclose 上游
  5. 任意 candidate 成功 → 返回 200 + _router metadata (provider/model/chain/latency/stale)
  6. 全失败 → 502 + 完整尝试链 metadata

4. 核心特性 (11 项)

① 多 Key 轮询

每 provider 配 N 个 key, 失败自动顺延. key_index 嵌入 _router metadata 便于排查

② 401 换 Key

当前 key 401 → 立刻换下一个 key 重试, 跳过该 key 当前请求. 配合 multi-key 池, 单 key 失效不阻塞

③ 5xx 跨 Provider 重试

5xx 错误跨 candidate 链重试, max_retry=2 + retry_backoff_ms [0, 500] 指数退避

④ 上下文桥接 (v3.4)

切链时注入 system message 同步上下文: "你正在接续, 前面 N 次尝试失败, partial 输出 X, 你的任务 1/2/3/4"

⑤ 切链 Race 防御 (v3.5)

stream 模式切链时显式 aclose() 上游 httpx, 防止旧模型的迟缓 reply 晚到错配新请求. smr_request_id 校验

⑥ 主动盘点 (v3.5)

POST /v1/admin/context_review 拿 smr_request_id 聚合报告: 参与模型/切链次数/原因/累积 partial/stale

⑦ 上下文窗口评分 (v3.8)

7 档 context bonus 4K=2 → 200K=20, 默认按模型能力排序. 可通过 config.classifier.context_window_bonus 自定义

⑧ 切链压缩 (v3.8)

切到 8K 模型但 body 1K tokens → 触发 paragraph_chunk (拆段不增 message) 或 history_trim (留 system + 最近 K 轮)

⑨ 公开 API (v3.7.1)

/v1/public/chat/completions 多租户, 独立 API key 管理 + model_filter 通配 (include/exclude/all)

⑩ Admin UI i18n (v3.6)

中/英双语切换, sidebar nav (Dashboard/Providers/Models/Keys/Stats/Config/Classifier/Version), 真实使用量统计卡片

⑪ 自升级 (v3.1)

GitHub release 1h 缓存 + /v1/admin/upgrade 生成 git/pip/docker/binary 4 种命令 + warning 风险提示, 不直接执行

6. 关键设计决策

决策理由拒绝方案
OpenAI 协议 100% 兼容现有 SDK / LangChain / curl 0 改动接入自定义协议 (迁移成本高)
Docker 化部署老大主机常重启, docker 隔离可移植systemd 直跑 (依赖难管理)
config.yaml 外挂 + 热重载改 key/路由无需重启容器, 5s watcher环境变量 (不能动态改)
state/ volume 挂载容器重启 = 不丢 public_keys/penaltiesin-memory (重启即丢)
Host 模式部署 (老 v3.7.1)跟主机共享 network, 简单 (但有端口冲突风险)bridge + port mapping
bridge + 6475:6473 (新 v3.8.0)端口映射清晰, 跟老实例隔离覆盖老端口 (中断生产)
不替老大改 docker 镜像老大 control deploy 时机, echo 跑通验证即可直接覆盖 :6473 容器
GitHub release 缓存 1h减少 API 调用 + 抗 rate limit每次请求实时拉
升级命令只生成不执行SMR 不直接做危险操作, owner 手动执行auto restart (破坏性)
占位符用 <X_API_KEY_PLACEHOLDER>避开 GH secret scanning 误判${X_API_KEY} (被 GH 拦)
