## 1. 背景 (Why)

### 1.1 老大原话复述
> "smr 每次切换模型的时候, 也同步上下文和任务给新的模型, 不然对话连不上. 同时要判断回传的信息是否过期, 已过期的话, 加上过期标记."

### 1.2 痛点技术分析
| # | 痛点 | coder 视角分析 |
|---|---|---|
| 1 | chain rotation 切到新 candidate 时, 新模型只看到原始 prompt | 上下文丢失导致上下文断层，需桥接 |
| 2 | 新模型重复输出前面已说过的内容 | 缺乏上下文状态，重复概率高 |
| 3 | 请求耗时 > 30 min 时, 新模型不知道信息可能已过期 | 时间戳管理需精确， stale 判定关键 |

### 1.3 设计目标 (coder 评估)
SMR v3.4.0 实现 3 大机制：
1. **非流式切换** → ContextBridge 注入 system prompt
2. **流式切换** → SSE sentinel `_smr_bridge` 协议
3. **全场景** → stale 计时敏感

## 3. 依赖图 + 单例 (coder 视角)

### 3.1 为什么 context_bridge.py 独立？
**跟 echo 不同**: echo 可能认为这只是集成，coder 看作**架构分层**:
- `context_bridge.py` 职责单一：上下文桥接
- `openai_routes.py` 职责：路由 + 协议处理
- 解耦后测试友好 (单独 unit test context_bridge)

### 3.2 单例模式 (coder 思考)
| 组件 | 形态 | 为什么 |
|---|---|---|
| ContextBridge | lifespan 实例化 + init() 注入 | **跟 v3.3 Registry/Engine 一致**，依赖注入优于全局 |
| SwitchHistory | deque(maxlen=5) | **自动裁剪无需手动**，比 list + 裁剪更安全 |

**跟 echo 不同**: echo 可能不强调 deque 为什么选型，coder 认为这是**并发安全**考量。

## 4. API 形态 (coder 视角)

### 4.1 函数设计原则
```python
bridge.inject_into_messages(messages, history)  # in-place 修改
bridge.get_stats()  # 独立函数 vs /stats endpoint
```

**coder 观点**: 
- in-place 好处：减少拷贝，性能优
- 但**不返回新列表**，有风险（副作用）
- `get_stats()` 作为函数**更灵活**，admin endpoint 可以封装调用

### 4.2 SSE sentinel 协议
```
data: {"_smr_bridge": {...}}
```
**coder 评估**: 
- 标准 SSE 兼容，老客户端可降级
- 客户端解析需健壮（忽略不可识别 data）
- **缺少 schema 版本**，后续扩展需注意

## 5. 类型设计 (coder 视角)

### 5.1 SwitchRecord 用 `@dataclass` vs Pydantic
| 选项 | 优点 | 缺点 |
|---|---|---|
| @dataclass | 不可变、轻量、静态分析好 | 无验证 |
| Pydantic | 自动验证、序列化好 | 重、运行时开销 |

**coder 建议**: 用 `@dataclass` 因为：
- 数据只写一次，后续不变
- 静态类型检查已覆盖大部分
- **性能敏感**场景（路由热路径）

**跟 echo 不同**: echo 可能考虑 Pydantic 的便捷，coder 优先性能。

### 5.2 LSP 假报
`Any` type hint 解：
- **是 hack**，但 v3.3 同型处理 (init 时填充) 一致
- 更稳方案: `Protocol` + `TYPE_CHECKING` 隔离运行时检查
