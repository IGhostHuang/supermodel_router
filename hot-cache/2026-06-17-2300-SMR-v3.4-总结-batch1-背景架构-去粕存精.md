## 1. 背景 (Why)

### 1.1 老大原话 (2026-06-17 22:00)

> "smr 每次切换模型的时候, 也同步上下文和任务给新的模型, 不然对话连不上. 同时要判断回传的信息是否过期, 已过期的话, 加上过期标记."

### 1.2 痛点 3 个 (echo + coder 合并)

| # | 痛点 | 技术分析 (coder) |
|---|---|---|
| 1 | chain rotation 切到新 candidate 时, 新模型只看到原始 prompt | **上下文丢失导致上下文断层**, 需桥接 |
| 2 | 新模型重复输出前面已说过的内容 | **缺乏上下文状态**, 重复概率高 |
| 3 | 请求耗时 > 30 min 时, 新模型不知道信息可能已过期 | **时间戳管理需精确**, stale 判定关键 |

### 1.3 设计目标 (3 大机制)

1. **非流式切换** → 注入"上下文桥接"system prompt
2. **流式切换** → 切到新 candidate 第一个 chunk 前发 `_smr_bridge` SSE sentinel
3. **全场景** → 整个请求耗时 > 阈值时标 `stale=true`

### 1.4 适用范围 (echo 独有, 保留)

- **SMR 是 OpenAI 兼容的多 provider LLM 路由网关** (前 FMR / free-model-router)
- v3.4 之前已有: 多 provider / 多 key 真正轮询 / penalty 降分 / 4 模式过滤 / 智能路由
- v3.4 新增层: chain rotation 切换时**额外做"上下文桥接"**, 跟切换逻辑解耦
- 不破坏现有 chain rotation / penalty / 路由逻辑

## 3. 依赖图 (echo + coder 合并)

```
config.yaml
    ↓ load
config.py (单例 config)
    ↓ 读 context_bridge 段
context_bridge.py (ContextBridge 单例, 职责: 上下文桥接)
    ↓ inject
openai_routes.py (init 三参: registry + engine + bridge, 职责: 路由 + 协议)
    ↓ lifespan
app.py (lifespan 中实例化, 注入)
    ↓ 暴露
admin_api.py (init 五参, 跟 v3.3 风格一致)

数据流 (一次请求):
client → openai_routes.chat_completions()
  → engine.execute_chain()
    → 候选 A 失败 → SwitchRecord 入 history (deque 自动滚动)
    → 候选 B 调用 → bridge.inject_into_messages(messages, history) ← 关键
    → 候选 B 响应 → bridge.is_stale(age) → _router.stale / age_seconds 字段
  → 返回响应 (流式: 切到 B 第一个 chunk 前 yield _smr_bridge sentinel)
```

**架构分层 (coder 强调, echo 未提)**:
- `context_bridge.py` 职责单一: 上下文桥接
- `openai_routes.py` 职责: 路由 + 协议处理
- 解耦后**测试友好** (单独 unit test context_bridge)

## 4. 单例 vs 注入 (coder 加固)

| 组件 | 形态 | 为什么 (coder) |
|---|---|---|
| `config` | 模块单例 | 已有, 不动 |
| `ContextBridge` | lifespan 实例化 + `init()` 注入 | 跟 v3.3 Registry/Engine **一致**, **依赖注入优于全局单例** |
| `SwitchHistory` | `deque(maxlen=5)` 内部属性 | **自动裁剪无需手动**, 比 list + 裁剪更安全 (并发友好) |
| `SwitchRecord` | `@dataclass` 不可变 | 数据只写一次, 静态类型好, 性能敏感场景适合 (vs Pydantic 重) |

## 5. API 形态 (echo + coder 合并)

| 入口 | 形态 | coder 评估 |
|---|---|---|
| 配置 | `config.yaml` 加 `context_bridge` 段 | 5 字段 (enabled/threshold/max_history/sentinel_enabled/inject_template) |
| 调用 | `bridge.inject_into_messages(messages, history)` | **in-place 修改** (减少拷贝, 性能优) 但**有副作用风险** (调用方需知) |
| 查询 | `bridge.get_stats() / get_config() / is_stale(age)` | 独立函数**更灵活**, admin endpoint 封装调用即可 |
| 管理 | `GET/PUT/POST /v1/admin/context_bridge` | 3 endpoint 略多, 但**跟 v3.3 风格一致** (config/penalty/decay/upgrade 分 endpoint) |

**SSE sentinel 协议** (coder 评估):
- `data: {"_smr_bridge": {...}}` 标准 SSE 兼容, 老客户端可降级
- 客户端解析需健壮 (忽略不可识别 data)
- ⚠️ **缺少 schema 版本字段** (echo 加了 `version=3.4.0`, 但只在 bridge 单例配置里, 建议 v3.5 加到 sentinel 顶层)
