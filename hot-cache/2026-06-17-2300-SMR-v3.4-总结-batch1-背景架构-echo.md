## 1. 背景 (Why)

### 1.1 老大原话 (2026-06-17 22:00)

> "smr 每次切换模型的时候, 也同步上下文和任务给新的模型, 不然对话连不上. 同时要判断回传的信息是否过期, 已过期的话, 加上过期标记."

### 1.2 痛点 3 个

| # | 痛点 | 后果 |
|---|---|---|
| 1 | chain rotation 切到新 candidate 时, 新模型只看到原始 prompt | 不知道前面模型为什么失败 / 部分响应, "对话断层" |
| 2 | 新模型重复输出前面已说过的内容 | 用户体验差 (重复啰嗦) |
| 3 | 请求耗时 > 30 min 时, 新模型不知道信息可能已过期 | 还在用"最新"口吻回答过时事实 |

### 1.3 设计目标

SMR v3.4.0 实施 **3 大机制** (batch 2 详解):
1. **非流式切换** → 注入"上下文桥接"system prompt
2. **流式切换** → 切到新 candidate 第一个 chunk 前发 `_smr_bridge` SSE sentinel
3. **全场景** → 整个请求耗时 > 阈值时标 `stale=true`

### 1.4 适用范围

- **SMR 是 OpenAI 兼容的多 provider LLM 路由网关** (前 FMR / free-model-router)
- v3.4 之前已有: 多 provider / 多 key 真正轮询 / penalty 降分 / 4 模式过滤 / 智能路由
- v3.4 新增层: chain rotation 切换时**额外做"上下文桥接"**, 跟切换逻辑解耦
- 不破坏现有 chain rotation / penalty / 路由逻辑

### 2.2 模块依赖图

```
config.yaml
    ↓ load
config.py (单例 config)
    ↓ 读 context_bridge 段
context_bridge.py (ContextBridge 单例)
    ↓ inject
openai_routes.py (init 三参: registry + engine + bridge)
    ↓ lifespan
app.py (lifespan 中实例化, 注入)
    ↓ 暴露
admin_api.py (init 五参: registry + engine + model_manager + start_time + bridge)

数据流 (一次请求):
client → openai_routes.chat_completions()
  → engine.execute_chain()
    → 候选 A 失败 → SwitchRecord 入 history
    → 候选 B 调用 → bridge.inject_into_messages(messages, history) ← 关键
    → 候选 B 响应 → bridge.is_stale(age) → _router.stale / age_seconds 字段
  → 返回响应 (流式: 切到 B 第一个 chunk 前 yield _smr_bridge sentinel)
```

### 2.3 单例 vs 全局

| 组件 | 形态 | 原因 |
|---|---|---|
| `config` | 模块单例 (from .config import config) | 已有, 不动 |
| `ContextBridge` | lifespan 实例化 + 模块全局 (init() 注入) | 跟 Registry/Engine 同样模式, 跟 v3.3 一致 |
| `SwitchHistory` | ContextBridge 内部属性 (deque) | 自动滚动, 无需外部状态 |
| `SwitchRecord` | dataclass, 不可变 | 写入即定型, 防止异步竞态 |

### 2.4 API 形态

| 入口 | 形态 | 说明 |
|---|---|---|
| 配置 | `config.yaml` 加 `context_bridge` 段 | 5 字段 (enabled/threshold/max_history/sentinel_enabled/inject_template) |
| 调用 | `bridge.inject_into_messages(messages, history)` | 在 openai_routes 切链时调用, 修改 messages 列表 |
| 查询 | `bridge.is_stale(age_seconds) / age_seconds(start_time) / get_stats() / get_config()` | 状态查询, 不修改状态 |
| 管理 | `GET/PUT/POST /v1/admin/context_bridge` | 热更新配置 + 清零 stats |

### 2.5 跟 v3.3 的边界 (不破坏)

✅ **不动**:
- chain rotation 逻辑 (`engine.execute_chain` 主体)
- penalty 降分机制 (`engine_stats.json`)
- 路由策略 (provider/model 排序)
- `/v1/chat/completions` 协议 (OpenAI 兼容)
- `/v1/models` 拉取阶段
- 多 key 轮询

✅ **新增** (叠加层):
- context_bridge 段配置 (默认 enabled=true)
- `_router` 字段扩展 (4 新字段, 不影响老客户端解析)
- 流式首 chunk 前的 SSE sentinel (走标准 SSE 协议, 老客户端忽略 data: 字段不识别可降级)
